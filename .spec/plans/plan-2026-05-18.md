## 4. Task C2 — Decode `id` & `role` + Field Proposer di Payload Submit

### Konteks

Fitur US-09 menambahkan informasi **pengusul** (proposer) ke form Imah Aing. Data ini berasal dari:

1. **Encoded `meta` query param** — sekarang menyertakan field `id` dan `role` (opsional). **BE sudah konfirmasi** field baru diterima.
2. **Logika fallback**: jika `role === 'warga'`, data pengusul diambil dari `state.dataPengusul` (yang diisi user). Jika `role` selain `'warga'`, data diambil dari decoded meta param. **Product konfirmasi**: hanya role `'warga'` yang perlu penanganan khusus, tidak ada role lain.

  if (!data || typeof data !== 'object') {
    return

**C2.3 — Tambah helper `resolveProposerField` untuk logika fallback**

Buat function baru di scope modul store:

```js
/**
 * Resolve field proposer dengan aturan:
 * - Jika meta.role selain 'warga' → pakai nilai dari meta (state.dataPengusul)
 * - Jika meta.role === 'warga' atau kosong → pakai nilai dari state.dataPengusul (diisi user)
 */
function resolveProposerField(state, fieldName) {
  const role = (state.dataPengusul.role || '').trim().toLowerCase()
  if (role && role !== 'warga') {
    return state.dataPengusul[fieldName] || ''
  }
  return state.dataPengusul[fieldName] || ''
}
```

> Catatan: karena nama field di meta (`name`, `phone`, `email`) sama dengan nama field di `dataPengusul`, helper cukup menerima satu `fieldName`. Logika fallback-nya adalah: non-warga → nilai dari meta (yang sudah masuk ke `dataPengusul`), warga/kosong → nilai dari `dataPengusul` (diisi user). Secara praktis, keduanya membaca dari `state.dataPengusul[fieldName]` — perbedaannya hanya eksplisit untuk dokumentasi dan future-proofing kalau suatu saat field proposer dipisah.

**C2.4 — Tambah field ke payload `submitForm`**

## 9. Catatan untuk Junior Developer

- **C1.1**: `penyebabDetailMap` ditaruh di `data()` sebagai objek biasa (bukan computed) karena nilainya statis. `selectedPenyebabDetail` pakai computed agar reaktif terhadap perubahan `kondisiRumah.penyebabKerusakan`.
- **C1.2**: Info box disisipkan **setelah** `</ValidationProvider>` penutup dropdown, **sebelum** `ValidationProvider` deskripsi kondisi. Jangan taruh di dalam `ValidationProvider` karena bukan bagian validasi.
- **C1.2 CSS**: Kalau pakai Tailwind utility + `<transition>`, tidak perlu tulis `@keyframes` manual. Kalau pakai CSS murni, tambahkan di `<style scoped>`.
- **C1 — `buildImahAingDescription` TIDAK diubah**. Product sudah konfirmasi field `description` tidak perlu perubahan.
- **C2.1**: Tambah field ke `getDefaultState()` — `RESET_STATE` sudah generik (iterasi `Object.keys(def)`), otomatis ikut.
- **C2.2**: `setField` internal sudah skip nilai kosong. Tidak perlu guard tambahan.
- **C2.3**: `resolveProposerField` hanya membaca `state`, tidak menulis. Panggil dari dalam action `submitForm`.
- **C2.4**: Field `user_*` existing **jangan dihapus** — BE konfirmasi tetap wajib. Field `proposer_*` adalah tambahan.
- **Value `role` case-insensitive**: gunakan `.trim().toLowerCase()` saat membandingkan dengan `'warga'`.
- **C4**: Hati-hati saat menghapus blok mock. Pastikan kurung kurawal `}` dan indentasi untuk blok pemanggilan API yang asli diperbaiki dengan rapi.
